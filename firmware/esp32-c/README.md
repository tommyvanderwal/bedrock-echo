# Bedrock Echo witness — ESP32-POE-ISO firmware (C / ESP-IDF)

Third reference implementation of the BEW1 protocol, alongside the Rust
witness and the Python reference. Targets the **Olimex ESP32-POE-ISO**
(ESP32 + LAN8710A Ethernet PHY over RMII with 50 MHz ref clock generated
by the ESP32's APLL on GPIO17).

## What this ports across

The BEW1 protocol is defined byte-for-byte in `PROTOCOL.md` at the repo
root, with checked-in test vectors in `testvectors/`. The C impl here
reuses those vectors for interop validation: encode + decode match the
Python/Rust impls exactly.

```
    ├── bew1_eth.c          # Olimex POE-ISO Ethernet bring-up
    ├── bew1_key.c          # NVS load-or-generate of the X25519 private key
    └── main.c              # init, event loop, UDP task
```

## Build + flash

```sh
. ~/esp/esp-idf/export.sh
cd firmware/esp32-c
idf.py set-target esp32
idf.py build
idf.py -p /dev/ttyUSB0 flash monitor
```

First boot generates a fresh X25519 keypair and writes the private key to
NVS. Subsequent boots read the key back. The **only thing persisted**;
everything else (cluster key table, node table) lives in RAM per the
design (PROTOCOL.md §1).

## Board-specific notes (Olimex ESP32-POE-ISO)

- **PHY**: LAN8710A at address 0, driver `lan87xx` (register-compatible).
- **RMII ref clock**: generated by ESP32 APLL, output on **GPIO17**
  (`EMAC_CLK_OUT` mode).
- **PHY power**: **GPIO12** — driven HIGH after boot to enable the PHY's
  on-board power supply. GPIO12 is also an ESP32 strapping pin that
  selects flash voltage at reset; the Olimex board has an external
  pull-down to keep it LOW at power-on, and we only drive it HIGH after
  the main app has started — so this is safe without eFuse tweaks.
- **MDC/MDIO**: GPIO23 / GPIO18.
- **RMII data/clk pins**: fixed by ESP32 silicon (TXD0=19, TXD1=22,
  TX_EN=21, RXD0=25, RXD1=26, CRS_DV=27).
- **LAN8710A post-power delay**: ~30 ms before first MDIO transaction
  (empirical; datasheet says ≥100 µs).

## Crypto

- **X25519**: embedded TweetNaCl scalar-mult (~80 lines, public domain).
  Chosen over mbedTLS ECP because `mbedtls_ecp_mul` on Curve25519 depends
  on config knobs that vary between mbedTLS builds. The TweetNaCl routine
  is a single self-contained function with no external dependencies.
- **HMAC-SHA256 / HKDF-SHA256 / ChaCha20-Poly1305**: mbedTLS (already in
  the IDF component tree; no extra dependencies).
- **Randomness**: `esp_fill_random()` — ESP32's on-chip RNG, available
  without Wi-Fi/BT init.

## Interop status

- **Python client → C witness**: 3/3 `test_interop_live.py` tests PASS.
  Same Python test suite used for the Rust witness works unchanged.
- **Node daemons (Python, on the KVM testbed) → ESP32 witness**: full
  9-scenario harness PASS. No split-brain under any failure mode.
- **Hardware reset of ESP32 witness** (via `esptool.py`, simulating
  power-cycle): nodes see UNKNOWN_SOURCE → auto-rebootstrap → back in
  normal operation within ~10 seconds. Invariant held throughout.

## Binary size

```
Total image: ~306 KB (including bootloader + partition table)
  Free flash: 4 MB - ~310 KB = ~3.7 MB unused
  Free RAM:   ~350 KB (IRAM + DRAM, mostly heap headroom)
```

Well within headroom for ESP32. No optimisations beyond `-Os`; no LTO yet.
